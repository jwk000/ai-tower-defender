# Tower Defender — 统一词条系统设计

> 当前版本：**v1.1（2026-05-21）**
> 本文档定义统一词条（Modifier）系统的设计目标、来源分类、出现流程、表现规范、规则边界与配置接口。
> 词条效果定位上接近遗物：都是 Run 内持续或半持续生效的被动规则改写；区别只在于**来源、获取时机、持续范围**。

---

## 1. 系统定位

### 1.1 一句话定义

**Modifier（词条）是一个统一的“被动规则改写来源”；它可以来自关卡、秘境或遗物，并在指定持续范围内生效。**

### 1.2 三类来源

| 来源 | sourceType | 获得方式 | 常见持续范围 |
|------|------------|---------|-------------|
| **关卡词条** | `level_modifier` | 进入关卡前从该关池中随机 | `current_level` |
| **秘境词条** | `sanctum_modifier` | 秘境事件、祭坛、交换、风险奖励 | `current_level` / `next_level` / `next_n_levels` / `run` |
| **遗物词条** | `relic` | 关后奖励 / 商店 / 事件 | 通常 `run` |

### 1.3 设计目标

| 目标 | 说明 |
|------|------|
| **增强关卡重玩性** | 同一关因词条不同，构筑判断与站位重点会变化 |
| **在关外制造预判决策** | 玩家在看到词条后，再决定这一关更偏向保守、防守还是爆发 |
| **放大已有构筑差异** | 词条不创造全新系统，而是优先放大已有单位/流派/敌人机制 |
| **承担轻量“变体规则”职责** | 比遗物更短周期，比天气更强规则，形成单关记忆点 |
| **不替代关卡主题** | 词条是关卡主题上的“第二层变化”，不能盖过关卡本身 signature flavor |

### 1.3 与遗物、天气、关卡 Twist 的区别

| 系统 | 来源 | 作用时长 | 典型效果 |
|------|------|---------|---------|
| **遗物词条** | 奖励、商店、事件 | 通常跨后续多关持续 | 强化玩家整个 Run 的某种策略轴 |
| **秘境词条** | 秘境事件 | 1 关 / 数关 / 整局 | 风险收益型临时规则改写 |
| **天气** | 关卡 YAML / 场景系统 | 当前关卡 | 偏表现+轻规则，如射程 -20% |
| **关卡 Twist** | 关卡固有设计 | 当前关卡 | 该关的主机制，如钻地段免疫地面 AoE |
| **关卡词条** | 进入关卡前随机 | 当前关卡 | 围绕单位/敌人/部署/经济的小到中强度规则变体 |

**边界原则**：
- 遗物词条是“我这局拿到了什么长期强化”。
- 秘境词条是“我为了收益接受了什么阶段性规则变化”。
- 关卡词条是“这关临时发生了什么”。
- 天气是“这关环境怎样”。
- Twist 是“这关核心考题是什么”。

## 2. 核心流程

### 2.1 关卡词条流程

```text
上一关结算完成
  ↓
进入下一关前，确定目标关卡
  ↓
根据目标关卡配置的 modifierPool 随机 1 个关卡词条
  ↓
弹出“本关词条揭示界面”
  ↓
玩家确认后进入战斗
  ↓
关卡词条本关生效
  ↓
关卡结束后失效
```

### 2.2 秘境词条流程

```text
进入秘境
  ↓
触发事件 / 做出选择
  ↓
获得一个 sanctum_modifier
  ↓
写入当前 Run 状态
  ↓
按其 activeScope 在当前关 / 下一关 / 后续 N 关 / 整局生效
```

### 2.3 遗物词条流程

```text
关后奖励 / 商店 / 事件获得遗物
  ↓
写入当前 Run 状态
  ↓
后续进入每一关时自动参与结算
  ↓
Run 结束后统一失效
```

### 2.2 随机规则

- **每关固定随机 1 个词条**
- 默认从该关 `modifierPool` 中**等权随机**
- 若后续需要，可支持权重 `weight`
- 同一次 Run 中，**不同关卡允许抽到同类方向词条**，但不应连续 3 关出现几乎同义词条
- 同一关的词条池建议 **3–5 个候选**，避免过散或过单薄

### 2.3 可见性规则

- 词条必须在**进入关卡前明确揭示**，不能隐性生效
- 进入战斗后，词条名称与简述必须常驻于**顶部 HUD**
- 玩家可在战斗内通过悬停 / 点击 HUD 词条区域查看详情

## 3. 词条设计原则

### 3.1 强度原则

1. **单词条只改变一个主判断轴**
   - 例如：强化某类单位、削弱某类单位、改变某种部署偏好、改变某类敌人的威胁
2. **词条强度低于 Boss 机制，高于天气提示**
3. **词条不能直接废掉玩家整个构筑**
   - 可以针对、压制，但不能让某一主流派“完全无法玩”
4. **词条优先提供“偏置”，少做“封杀”**
   - 优先 `+30% / -20% / 额外触发一次 / 成本+1`
   - 谨慎使用 `无法部署 / 完全免疫 / 永久沉默`

### 3.2 内容原则

优先做以下四类词条：

| 类别 | 说明 | 例子 |
|------|------|------|
| **单位强化型** | 强化我方某类单位或卡牌标签 | 箭塔攻速提升、士兵 HP 提升 |
| **单位抑制型** | 削弱我方某类单位或卡牌标签 | 法术耗能 +1、陷阱充能变慢 |
| **敌军变体型** | 强化敌方某类表现，使该关解题变化 | 飞行敌人速度提升、精英护甲提升 |
| **战场规则型** | 小幅改变部署与节奏规则 | 前 20 秒能量回复更快、首个塔成本降低 |

### 3.3 适配原则

- 词条必须与关卡主题 / enemy roster / flavor **有语义关联**
- 优先围绕该关推荐应对方式做“顺拧”或“反拧”
- 不做完全脱离关卡主题的随机词条
  - 例：雪地关突然出现“海盗单位全部隐形”不合适

## 4. 词条分类

### 4.1 按受影响对象分类

| 分类 | 作用对象 | 示例 |
|------|---------|------|
| `tower_buff` | 塔 | “箭塔伤害 +25%” |
| `soldier_buff` | 士兵 | “士兵部署后前 5 秒获得护盾” |
| `trap_buff` | 陷阱 | “陷阱首次触发必定暴击” |
| `spell_debuff` | 法术 | “法术能量消耗 +1” |
| `enemy_buff` | 敌军 | “飞行敌人移动速度 +20%” |
| `economy_rule` | 关内资源 | “初始能量 +2” |
| `deploy_rule` | 部署规则 | “首个部署的士兵返还 1 能量” |

### 4.2 按效果方向分类

| 方向 | 含义 |
|------|------|
| **正面词条** | 明显偏向帮助玩家 |
| **负面词条** | 明显偏向增加挑战 |
| **摇摆词条** | 同时有利有弊，要求玩家调整打法 |

设计建议：
- 前 1–2 关以**正面 / 轻摇摆词条**为主
- 中段关卡加入更多**摇摆 / 轻负面词条**
- Boss 关词条以**高识别、低歧义**为主，避免过度复杂

## 5. 推荐词条模板

### 5.1 强化型模板

- **火力校准**：所有箭塔/弩塔攻击速度 +25%
- **战地动员**：士兵部署费用 -1（最低为 1）
- **连锁超载**：电塔链式弹跳次数 +1
- **工程预热**：本关开始时初始能量 +2

### 5.2 削弱型模板

- **法力紊乱**：所有法术牌能量消耗 +1
- **潮湿火药**：炮塔与导弹塔攻击速度 -20%
- **地表塌陷**：陷阱触发后额外延长 20% 再装填时间
- **士气低落**：士兵最大生命值 -20%

### 5.3 摇摆型模板

- **孤注一掷**：塔攻击力 +30%，但士兵生命值 -25%
- **速攻号令**：前 20 秒能量回复速度 +100%，之后恢复正常
- **谨慎推进**：首个部署的塔免费，但后续塔牌费用 +1
- **猎杀高威胁**：对精英/Boss 伤害 +25%，对杂兵伤害 -15%

## 6. 与遗物系统的关系

### 6.1 共同点

- 都是 **RuleModifier / PassiveEffect** 风格的被动效果
- 都适合复用统一配置驱动表达：
  - `targetTags`
  - `modifiers`
  - `triggers`
  - `durationScope`

### 6.2 统一抽象

实现层面应统一为：
- 词条不另起一套效果 DSL
- 遗物不保留独立效果执行分支
- 所有来源都作为一类 `PassiveSource`
- 区别仅在于：
  - `sourceType`
  - `activeScope`
  - `grantedAt`

```typescript
interface PassiveSource {
  sourceType: 'level_modifier' | 'sanctum_modifier' | 'relic'
  sourceId: string
  activeScope: 'current_level' | 'next_level' | 'next_n_levels' | 'run'
}
```

### 6.3 统一后的收益

- 同一效果只写一份，可被多来源复用
- 新增秘境词条时几乎等于“新增一个新的 source 包装”
- 词条、遗物、秘境效果叠加时有统一优先级与结算入口
- UI 可按来源分类展示，但规则层无需区分三套逻辑

## 7. 与关卡系统的关系

### 7.1 关卡配置新增字段

```yaml
levelId: level_03
name: 极地暴雪要塞
modifierPool:
  - lvmod_blizzard_supply
  - lvmod_frozen_bolts
  - lvmod_shivering_ranks
```

### 7.2 词条池设计规则

每关的 `modifierPool` 应满足：
- 至少 **3 个**词条候选
- 最多建议 **5 个**，避免记忆负担过高
- 至少包含：
  - 1 个偏正面
  - 1 个偏负面
  - 1 个摇摆型
- 词条必须与该关主题/敌人/机制对应

### 7.3 关卡与词条的组合边界

- **Twist 是必然题，词条是随机题**
- 词条不能覆盖或否定关卡 Twist
- 若某词条会与本关 Twist 产生极端组合，应在配置层显式禁配

示例：
- 关 2 已有“钻地段地面 AoE 失效”
- 则不应再叠一个“所有非探测塔攻击地下单位无效”的极端词条

## 8. UI / UX 需求

### 8.1 入场词条揭示界面

进入关卡时，在正式开战前弹出一张居中模态卡：

**展示内容**：
- 词条名
- 词条图标
- 1 句 flavor text
- 1–2 条明确效果说明
- “本关生效”标签
- 确认按钮：`进入关卡`

目标：
- 3 秒内读懂
- 让玩家在开战前形成“这关我要怎么打”的预期

### 8.2 HUD 常驻展示

战斗中顶部 HUD 新增词条区：

```text
[词条] 火力校准：箭塔/弩塔攻速 +25%
```

规则：
- 默认常驻单行简述
- 悬停 / 点击可展开详情
- 若词条为负面，采用偏橙/红警示色
- 若词条为正面，采用偏青/金强化色
- 若为摇摆词条，采用中性紫/白色

### 8.3 信息优先级

- HUD 词条信息优先级 **低于** 能量 / 波次 / 水晶 HP
- 但优先级 **高于** 纯装饰信息
- 不可藏进二级菜单；词条是本关核心规则之一

## 9. 配置接口（概念）

### 9.1 词条配置结构

```typescript
interface LevelModifierConfig {
  id: string;
  name: string;
  category:
    | 'tower_buff'
    | 'soldier_buff'
    | 'trap_buff'
    | 'spell_debuff'
    | 'enemy_buff'
    | 'economy_rule'
    | 'deploy_rule';
  polarity: 'positive' | 'negative' | 'swing';
  icon: string;
  flavorText: string;
  shortDescription: string;
  description: string[];
  effectRefs: string[]; // 复用遗物/被动效果定义
  tags: string[];
}
```

### 9.2 关卡配置结构

```typescript
interface LevelConfig {
  id: string;
  name: string;
  modifierPool?: string[];
}
```

### 9.3 运行时状态

```typescript
interface RunLevelState {
  currentLevelId: string;
  currentLevelModifierId?: string;
}
```

## 10. 结算与优先级

### 10.1 生效时机

- 进入关卡前随机并确定
- 进入战斗时注册到本关规则上下文
- 开战后直到关卡结束持续有效
- 离开关卡立即失效

### 10.2 与遗物叠加

建议统一采用“加算后乘算”或现有遗物规则优先级，不为词条单开特例。

推荐顺序：
1. 基础单位/卡牌原始值
2. 卡牌等级强化
3. 遗物修正
4. **词条修正**
5. 天气 / 关卡临时状态修正
6. 战斗中即时 Buff / Debuff

若项目现有遗物优先级已固定，则词条遵循同一框架，只新增 `sourceType` 区分来源。

### 10.3 冲突处理

- 同一属性被多个来源改写时，按既有被动系统规则合并
- 若某词条与某遗物效果完全冲突，以**最终结果可解释**为优先
- 避免设计“一个说可部署，一个说不可部署”的硬冲突词条

## 12. 不变式

1. **每关进入前随机且仅随机 1 个词条**
2. **词条必须在开战前明确展示**
3. **词条必须在战斗 HUD 常驻可见**
4. **词条默认仅对当前关卡生效**
5. **词条效果表达应复用遗物式被动效果框架**
6. **词条不能完全废掉玩家已有构筑**
7. **词条必须与该关主题/机制有关联**
8. **词条不能覆盖关卡 Twist 的核心存在感**
