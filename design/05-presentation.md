## 1. 渲染架构

当前运行时代码以 **Canvas 2D 命令缓冲** 为主渲染链路。`Renderer` 每帧清空命令队列，收集各系统写入的 `RenderCommand`，在 `endFrame()` 中按 `z` 从小到大稳定排序后绘制；同一 `z` 内保持写入顺序，因此实体可先按 Y 坐标排序再入队，得到稳定遮挡关系。

PixiJS 当前只保留 `Renderer.container: Container` 作为兼容桥接字段，主战斗画面、UI、天气和装饰物均由 Canvas 2D 绘制，不存在运行时多 `Container` 分层树。

```
Canvas viewport
 ├── Renderer.beginFrame()
 │    ├── viewport-space 深色背景
 │    └── 1920×1080 design-space transform
 ├── DecorationSystem
 │    ├── 天空/远景/云
 │    ├── 地图障碍复合几何
 │    ├── 草叶
 │    └── 飞鸟
 ├── RenderSystem
 │    ├── 地图瓦片/出生点/边框
 │    ├── 预警圈/地面伤痕
 │    ├── 实体复合外观
 │    ├── 状态/攻击/精英/Boss 表现
 │    └── 血条/冷却条/名字/徽记
 ├── post-render systems
 │    ├── 闪电、激光、地面伤痕补绘
 │    ├── UISystem
 │    └── CardEncyclopediaUI
 └── Renderer.endFrame()
      └── 按 RenderCommand.z 稳定排序绘制
```

`LayoutManager` 负责浏览器 viewport 到设计坐标的转换：设计分辨率固定为 1920×1080，缩放因子按 viewport 高度计算，设计画面横向居中；超宽屏会在左右留下额外可视空间。输入坐标通过 `toDesignX/toDesignY` 转回设计坐标后参与 UI 命中、拖卡和战场放置。

## 2. 层级系统

### 2.1 两层体系

层级系统分为 **逻辑层级** 和 **表现层级** 两个独立维度：

| 维度 | 逻辑层级 | 表现层级 |
|------|---------|---------|
| 所在域 | 棋盘空间高度 | 场景视觉远近 |
| 作用 | 决定攻击目标可达性 | 决定视觉遮挡关系 |
| 类比 | Z轴（上下） | Z-index（前后） |

### 2.2 逻辑层级（棋盘内5层）

```
Space（高空层）    — 最高，预留
LowAir（低空层）   — 飞行单位（蝗虫、蝙蝠、飞机、无人机）
Ground（地面层）   — 行走单位 + 塔 + 建筑 + 场景装饰物
Grid（地格层）     — 地面机关陷阱
Abyss（深渊层）    — 最低，预留
```

**层级交互规则** ：

| 规则 | 说明 |
|------|------|
| 低层可攻击高层 | Ground层单位可以攻击LowAir层单位 |
| 高层可攻击低层 | LowAir层单位可以攻击Ground层单位 |
| Grid可攻击Ground | 地刺对路过地面单位造成伤害 |
| 同层可攻击 | 同层单位可以互攻 |
| 弹道穿透 | 弹道飞行不受层级影响，由发射方和目标方层级决定可否命中 |

### 2.3 Canvas 命令渲染Z序（从后到前）

当前渲染不使用实体层容器，而是用 `RenderCommand.z` 排序。`RenderSystem` 的实体层映射为：

| z | 来源/内容 |
|---|----------|
| 0 | `RenderSystem.drawMap` 地图瓦片、出生点旗帜、场景边框 |
| 1 | 圆形弹坑外圈、部分地面标记 |
| 2 | 地面伤痕、毒雾光环、旧地格伤害点 |
| 3 | Abyss / BelowGrid 逻辑层实体 |
| 4 | AboveGrid 逻辑层实体，主要是机关；导弹预警圈也绘制在此层 |
| 5 | Ground 逻辑层实体，默认单位层 |
| 6 | LowAir 逻辑层实体；敌人攻击波纹提升到 Ground+1 |
| 7 | Space 逻辑层实体 |
| 8+ | 个别粒子、气泡、前景效果 |
| 20 | 选中塔/机关的范围预览 |

装饰系统在 `RenderSystem` 前写入命令，背景和静态装饰通常不设置显式 `z`，因此默认在 `z=5` 与地面实体同层；实际遮挡由写入顺序和实体 Y 排序共同决定。屏幕震动不独立成层，而是在 `RenderSystem.update()` 开始时对 Canvas context 追加平移。

### 2.4 Post-render 与 UI

`main.ts` 的 `onPostRender` 在 ECS 系统更新后执行：闪电、激光、地面伤痕补绘、`UISystem.renderUI()`、卡牌百科依次绘制。UI 仍通过 Canvas 2D/RenderCommand 与直接 Canvas 文本绘制完成，不使用单独 DOM 或 Pixi UI 容器。

## 3. 天气系统

### 3.1 天气类型

| 天气 | ID | 所属关卡 | 说明 |
|------|-----|---------|------|
| 晴天 | Sunny | 第2关 | 默认天气，阳光普照 |
| 下雨 | Rain | 第1关 | 大雨滂沱 |
| 下雪 | Snow | （预留） | 暴雪严寒 |
| 下雾 | Fog | （预留） | 浓雾弥漫，视野受限 |
| 夜晚 | Night | 第3、5关 | 夜幕降临 |
| 红雾 | RedMist | 第4关 | 赤红雾霾笼罩废土，远方火山将醒 |

### 3.2 视觉特效

| 天气 | 特效 |
|------|------|
| 晴天 | 左上角多层径向渐变太阳 + 5束斜向半透明阳光射线 + 低密度风线 |
| 下雨 | 双层斜雨：远层60滴、近层70滴；顶部乌云由28个大云团和40个小云团叠加；风线密度提高 |
| 下雪 | 40个固定槽位雪花（#ffffff），半径4-8px，初始横纵坐标随机分布，循环下落时从顶部重新进入，带正弦漂移 |
| 下雾 | 18个大雾团 + 30个小雾团，使用径向渐变软边、缓慢漂移和上下摆动 |
| 夜晚 | 70颗小星、35颗中星、8颗亮星 + 右上角三层光晕月亮 + 风线 + 暗角；可在关卡 `map.lighting.moonlight` 中启用棋盘外侧 bloom 和微光粒子 |
| 红雾 | 右上角巨大红色夕阳，使用外层红晕、暗红虹膜和黑色瞳孔构成“眼睛”剪影；天空 36 个暗红云/火山灰团快速横向移动；远方黑色山脉横贯天际，山顶 6 处红色闪光模拟即将喷发的火山 |

### 3.3 天气对游戏的影响

天气仅影响视觉氛围， **不影响战斗数值** 。

> 设计意图：避免玩家需要记忆天气对数值的影响，降低认知负担。天气纯粹用于烘托关卡氛围。

### 3.4 夜晚棋盘可读性

第3、5关使用夜晚天气时，棋盘主体必须保持可读。第5关深渊裂隙因主题色和夜晚遮罩叠加较暗，必须启用 `map.lighting.moonlight`：

- `enabled: true`
- `ambientAlpha: 0.16`，兼容旧配置；当前实现不使用该值绘制棋盘内部提亮层
- `bloomAlpha: 0.22`，用于棋盘向外 bloom 光晕和微光粒子强度

月光由 `BoardGlowSystem` 围绕 `RenderSystem.sceneOffsetX/Y/W/H` 绘制在棋盘外侧，只允许出现在棋盘外侧 bloom 区域，不进入棋盘内部。粒子必须保持稀疏、非均匀分布，不能沿棋盘边缘形成连续白色环形边框。月光不在棋盘内部叠加半透明铺色或径向遮罩，只通过外侧弱 bloom 营造被照亮的边缘氛围，不改变天气数值规则、单位属性或 UI 亮度。

**飞鸟** （所有关卡共通）：

| 属性 | 值 |
|------|-----|
| 形状 | 简化V形（2条短线相交于中心）+ 小三角头 |
| 大小 | 8-14px翼展 |
| 颜色 | `#ffffff`（白色） |
| 数量 | 2-5只/场景 |
| 运动 | 沿贝塞尔曲线飞行，翅膀正弦拍动（3-5Hz） |
| 行为 | 循环路径：左→右→飞出后从另一侧重入→随机改变高度 |
| 层级 | LowAir层（z=4） |

翅膀拍动周期（1周期=2次拍动/秒）：
- 0-25%：翅膀展开↑，身体+2px
- 25-50%：翅膀收起↓，身体-2px
- 50-75%：翅膀展开↑，身体+2px
- 75-100%：翅膀收起↓，身体-2px
- 翅膀角度：30°（收起）→120°（展开）正弦循环

| 类别 | 说明 | 实现方式 |
|------|------|---------|
| 静态装饰物增强 | 植物微动、火盆/紫火/熔岩口跳动 | `DecorationSystem` 每帧按时间偏移部件 |
| 动态环境生物 | 飞鸟、云朵、草叶 | `DecorationSystem` 内部状态数组驱动 |

## 5. 水晶视觉

水晶是玩家的核心防御目标，在地图左侧占据1格。

| 属性 | 值 |
|------|------|
| 形状 | 菱形复合几何体（2个菱形叠加，1主1辉光） |
| 主体色 | `#ff1744` 红色 |
| 辉光色 | `#ff5252`（外层，透明度0.4） |
| 大小 | 主体32px，辉光48px |
| 动画 | 上下浮动（±4px，正弦循环2秒周期） |
| 辉光呼吸 | 透明度从0.2→0.5→0.2循环（3秒周期） |
| 低血量(<30%) | 闪烁红色+呼吸频率加快至1.5秒周期 |
| 秒杀特效 | 水晶向目标方向发出红色光束+目标位置红色爆裂粒子 |
| 碎裂特效 | 水晶分裂为8-12个碎片→向上飘散→透明度渐变到0→完全消失（1.5秒） |

## 6. 程序化单位美术系统

### 6.1 目标

塔、士兵、机关、敌人、法术弹道全部使用统一的程序化美术语言。基础几何体不再直接作为最终外观，而是作为“身体/底座”的一层，叠加部件、光影、动作和状态材质形成完整角色。

本系统优先保持战斗可读性：

- 轮廓必须清楚，32px 左右的小单位仍能区分阵营和职责。
- 同类别单位共享形体语言，不同职责通过部件差异区分。
- 状态效果必须叠加在单位外观之上，不覆盖核心识别色。
- 所有外观只影响表现，不改变碰撞、寻路、攻击、数值。

### 6.2 配置来源

外观配置由两级来源组成：

| 来源 | 优先级 | 用途 |
|------|--------|------|
| YAML `visualParts` | 高 | 关键单位的手工定制外观 |
| `proceduralArt` 生成器 | 低 | 为未配置 `visualParts` 的所有单位生成默认复合外观 |

`visualParts` 采用通用复合部件结构：

- `bodyParts`：身体装饰、底座、护甲、炮管、纹章、裂纹等。
- `weapon`：士兵近战/远程武器和可挥砍部件。
- `bobStyle`：`walking`、`floating`、`static` 三种待机/移动动效。

### 6.3 类别形体语言

| 类别 | 主体语言 | 必备部件 | 动作 |
|------|----------|----------|------|
| 塔 | 稳定底座 + 职责装置 | 底座、核心、炮管/水晶/线圈/巢穴 | 待机呼吸，攻击时装置轻微强调 |
| 士兵 | 圆形/盾形身体 + 头盔/职业徽记 + 武器 | 头盔、职业徽记、武器或工具 | 行走 bob，攻击挥砍/举弓 |
| 机关 | 贴地结构 + 触发部件 | 地盘、齿、坑、风口、符文 | 触发时展开/脉冲 |
| 地面敌人 | 生物身体 + 威胁轮廓 + 职责部件 | 角/甲壳/爪/机械板/核心高光 | 行走 bob，攻击压缩前冲 |
| 飞行敌人 | 悬浮身体 + 翼/气囊/推进器 | 翅膀或推进器、阴影弱化 | floating bob |
| Boss | 最大主体 + 多层光环/重护甲/主题核心 | 至少12个复合部件：投影、双层光环、主体、发光核心、护甲、角/翼/炮管/虫足等专属部件 | 行走或悬浮 bob + 呼吸脉冲，二阶段变色、强轮廓 |
| 法术/弹道 | 几何轨迹 + 粒子 | 拖尾、内核、外环 | 生命周期内缩放/淡出 |

### 6.4 视觉层次

单个单位按以下顺序绘制：

1. 地面阴影或低空投影。
2. 光环/蓄力/毒雾等低层状态。
3. 主体基础形状。
4. 复合 `bodyParts`。
5. 武器、炮管、翅膀等前景部件。
6. 徽记、核心、高光。
7. 选中框、精英金边、Boss 阶段轮廓。

部件可通过 `offsetX/offsetY/size/h/color/alpha/stroke/strokeWidth/rotation` 控制。可移动单位的 `offsetX` 随 `facing` 翻转；塔和机关默认为静态不翻转。

### 6.5 品质规范

- 每个可见单位至少包含 3 个视觉层：主体、部件、高光/描边。
- 玩家单位以冷色和高饱和职业色为主，敌人以暖色、暗色或主题色为主。
- 机关必须贴地，不允许使用与士兵相同的面部语言。
- 飞行单位必须使用 `floating` 动效，并通过翅膀、气囊或推进器说明高度。
- 精英敌人保留金色轮廓，Boss 必须是全场最高复杂度单位：比普通敌人拥有更多几何层、更多描边/光环、主题专属结构，并且不得退化为单形状或静态外观。
- 法术弹道优先用程序化拖尾、圆环、闪光、粒子，不引入贴图依赖。

## 7. UI布局

### 7.1 总体布局（1920×1080）

```
┌─────────────────────────────────────────────────────┐
│ [💎HP] [💰金币]     [波次] [敌军] [天气]   [⏱] [▶][1x][⏸] │ ← 顶部HUD 36px
│                                                     │
│                                                     │
│                ┌───────────────┐                    │
│                │               │                    │
│                │   战场区域     │                    │
│                │  (1344×576)   │                    │
│                │               │                    │
│                └───────────────┘                    │
│                                                     │
│          ┌────┐ ┌────┐ ┌────┐ ┌────┐              │
│          │卡1 │ │卡2 │ │卡3 │ │卡4 │              │ ← 手牌区 800×180
│          └────┘ └────┘ └────┘ └────┘              │
└─────────────────────────────────────────────────────┘
```

战斗 UI 使用设计坐标绘制，顶部 HUD 背景条横跨当前 viewport 对应的 design-space 宽度；战场尺寸由关卡 `rows/cols/tileSize` 决定，常规 21×9×64 时为 1344×576，垂直居中在 36px 顶部 HUD 与底部 100px 面板预留区之间。

### 7.2 顶部HUD

| 元素 | 位置 | 内容 |
|------|------|------|
| 水晶HP | 左上起始位 | 💎图标 + 当前/最大HP |
| 金币数 | 水晶HP右侧 | 💰图标 + 数值 |
| 波次进度 | 顶部中部偏左 | "波次 3/7" |
| 敌军计数 | 波次进度右侧 | 当前存活敌军/本波已生成敌军 |
| 天气 | 敌军计数右侧 | 当前天气名称 |
| 倒计时 | 右侧按钮组左侧 | ⏱图标 + 秒数 |
| 右侧按钮组 | 顶部右侧 | 跳过倒计时/开始、倍速、暂停 |

顶部 HUD 条左右两侧必须保留对称页边距：水晶 HP 从 200px 内边距开始，金币位于水晶 HP 右侧并保持约 150px 横向间距；右侧按钮组最右边缘距离屏幕右边缘 200px，按钮组内部间距保持一致。

倒计时文字显示在右侧按钮组中的波次开始按钮左侧，文本右边缘与按钮左边缘保留约 12px 间距，避免与倍速/暂停按钮重叠。

### 7.7 关卡选择界面

关卡选择界面是玩家进入 Run 前的主题预览入口，背景必须随当前选中关卡切换，避免仅展示静态按钮列表。

| 元素 | 表现 |
|------|------|
| 主题背景 | 5 个关卡各有独立动态背景，使用 viewport 尺寸全屏绘制，程序化几何图形和粒子必须覆盖整个浏览器视口 |
| 默认选中 | 进入界面时默认选中第一个已解锁关卡 |
| 鼠标划过 | 鼠标移动到已解锁关卡卡片时立即选中该关卡，并切换对应主题背景 |
| 点击进入 | 点击已解锁关卡卡片立即进入该关卡 |
| 锁定关卡 | 可见但置灰，不响应选中和进入 |
| 选中反馈 | 当前选中卡片放大、描边增强，并显示主题摘要 |

## 8. 动画系统

### 8.1 单位动画

| 动画类型 | 说明 |
|---------|------|
| 复合外观呼吸 | `UnitAnimationSystem` 只推进 `Visual.partsId != 0` 的实体，`breathPhase` 以 2.2 rad/s 循环 |
| walking bob | `RenderSystem.drawUnitComposite` 中 Y 方向约 ±2px，X 方向轻微摇摆 |
| floating bob | Y 方向约 ±4px，无 X 摇摆 |
| static | 塔和机关不做行走 bob，仅保留约 1.5% 呼吸缩放 |
| 攻击 | 敌人使用 squash-stretch 与短距离 lunge；带武器的复合外观使用 `attackAnimTimer/Duration` 驱动武器摆动 |
| 受击 | `Visual.hitFlashTimer` 触发当帧白色闪烁 |
| 中毒/减速/冻结/眩晕 | 在实体颜色上叠加 tint，毒额外绘制绿色光环 |
| 精英/Boss | 精英敌人放大20%并使用金色描边；Boss 额外放大、二阶段红色混色/闪烁 |

### 8.2 弹道动画

| 弹道类型 | 表现 |
|---------|------|
| arrow | `Renderer` 的 arrow 形状：矩形箭杆 + 三角箭头，按目标点旋转 |
| 普通 projectile | 按 `Projectile` 组件位置绘制基础几何体，颜色/尺寸来自塔类型配置 |
| missile | 专用多层外观：橙红脉冲光晕、蓝色尾焰、黑色机身、红色菱形弹头，飞行角度来自 `Visual.idlePhase` |
| beam/laser | `LaserBeamSystem` 在 post-render 中补绘持续光束 |
| lightning | `LightningBoltSystem` 在 post-render 中补绘闪电线 |
| spell projectile | `SpellProjectileSystem` 管理法术弹道表现 |

### 8.3 粒子与屏幕效果

| 效果 | 触发时机 | 表现 |
|------|---------|------|
| 攻击波纹 | 敌人攻击动画期间 | 3个同心圆依次扩散并淡出 |
| 导弹预警 | 超级机器人导弹等 | 红色同心圆 + 十字准星，随 `TargetingMark.pulsePhase` 脉冲 |
| 弹坑/地面伤痕 | 导弹或地格伤害 | 深色圆形弹坑、边缘描边、裂纹和碎屑点 |
| 爆炸/死亡 | `ExplosionEffectSystem` / `DeathEffectSystem` | 生命周期内缩放、淡出 |
| 血粒子 | 受击或死亡 | `BloodParticleSystem` 驱动小粒子 |
| 屏幕震动 | 爆炸、Boss 技能等 | `ScreenShakeSystem.state` 被 `RenderSystem` 消费为 Canvas 平移 |
