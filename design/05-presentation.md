## 1. 渲染架构

当前运行时代码以 **Canvas 2D 命令缓冲** 为主渲染链路。`Renderer` 每帧清空命令队列，收集各系统写入的 `RenderCommand`，在 `endFrame()` 中按 `z` 从小到大稳定排序后绘制；同一 `z` 内保持写入顺序，因此实体可先按 Y 坐标排序再入队，得到稳定遮挡关系。

PixiJS 当前只保留 `Renderer.container: Container` 作为兼容桥接字段，主战斗画面、UI、天气和装饰物均由 Canvas 2D 绘制，不存在运行时多 `Container` 分层树。

```
Canvas viewport
 ├── Renderer.beginFrame()
 │    ├── viewport-space 深色背景
 │    └── 1920×1080 design-space transform
 ├── DecorationSystem
 │    ├── 天空/远景/云
 │    ├── 地图障碍复合几何
 │    ├── 草叶
 │    └── 飞鸟
 ├── RenderSystem
 │    ├── 地图瓦片/出生口/水晶端点/边框
 │    ├── 预警圈/地面伤痕
 │    ├── 实体复合外观
 │    ├── 状态/攻击/精英/Boss 表现
 │    └── 血条/冷却条/名字/徽记
 ├── post-render systems
 │    ├── 闪电、激光、地面伤痕补绘
 │    ├── UISystem
 │    └── CardEncyclopediaUI
 └── Renderer.endFrame()
      └── 按 RenderCommand.z 稳定排序绘制
```

`LayoutManager` 负责浏览器 viewport 到设计坐标的转换：设计分辨率固定为 1920×1080，缩放因子按 viewport 高度计算，设计画面横向居中；超宽屏会在左右留下额外可视空间。输入坐标通过 `toDesignX/toDesignY` 转回设计坐标后参与 UI 命中、拖卡和战场放置。

## 2. 层级系统

### 2.1 两层体系

层级系统分为 **逻辑层级** 和 **表现层级** 两个独立维度：

| 维度 | 逻辑层级 | 表现层级 |
|------|---------|---------|
| 所在域 | 棋盘空间高度 | 场景视觉远近 |
| 作用 | 决定攻击目标可达性 | 决定视觉遮挡关系 |
| 类比 | Z轴（上下） | Z-index（前后） |

### 2.2 逻辑层级（棋盘内5层）

```
Space（高空层）    — 最高，预留
LowAir（低空层）   — 飞行单位（蝗虫、蝙蝠、飞机、无人机）
Ground（地面层）   — 行走单位 + 塔 + 建筑 + 场景装饰物
Grid（地格层）     — 地面机关陷阱
Abyss（深渊层）    — 最低，预留
```

**层级交互规则** ：

| 规则 | 说明 |
|------|------|
| 低层可攻击高层 | Ground层单位可以攻击LowAir层单位 |
| 高层可攻击低层 | LowAir层单位可以攻击Ground层单位 |
| Grid可攻击Ground | 地刺对路过地面单位造成伤害 |
| 同层可攻击 | 同层单位可以互攻 |
| 弹道穿透 | 弹道飞行不受层级影响，由发射方和目标方层级决定可否命中 |

### 2.3 Canvas 命令渲染Z序（从后到前）

当前渲染不使用实体层容器，而是用 `RenderCommand.z` 排序。`RenderSystem` 的实体层映射为：

| z | 来源/内容 |
|---|----------|
| 0 | `RenderSystem.drawMap` 地图瓦片、出生口主体与循环 FX、水晶端点底图、场景边框 |
| 1 | 圆形弹坑外圈、部分地面标记 |
| 2 | 地面伤痕、毒雾光环、旧地格伤害点 |
| 3 | Abyss / BelowGrid 逻辑层实体 |
| 4 | AboveGrid 逻辑层实体，主要是机关；导弹预警圈也绘制在此层 |
| 5 | Ground 逻辑层实体，默认单位层 |
| 6 | LowAir 逻辑层实体；敌人攻击波纹提升到 Ground+1 |
| 7 | Space 逻辑层实体 |
| 8+ | 个别粒子、气泡、前景效果 |
| 30 | 棋盘覆盖层：选中塔/机关的范围预览、塔/单位 tips、升级/回收面板 |
| 100 | 普通 UI：顶部 HUD、手牌区、手牌详情 tooltip |
| 1000 | 全屏 UI：倒计时、暂停、胜负、抽卡、关间 Buff 选择 |

装饰系统在 `RenderSystem` 前写入命令，背景和静态装饰通常不设置显式 `z`，因此默认在 `z=5` 与地面实体同层；实际遮挡由写入顺序和实体 Y 排序共同决定。屏幕震动不独立成层，而是在 `RenderSystem.update()` 开始时对 Canvas context 追加平移。

棋盘上的单位与单位 tips 必须始终绘制在棋盘上方；所有普通 UI 必须绘制在棋盘覆盖层上方；全屏 UI 必须绘制在普通 UI 与棋盘覆盖层上方。因此塔升级面板属于棋盘覆盖层，会被抽卡、暂停、倒计时、Buff 选择等全屏 UI 覆盖。

### 2.4 Post-render 与 UI

`main.ts` 的 `onPostRender` 在 ECS 系统更新后执行：闪电、激光、地面伤痕补绘、`UISystem.renderUI()`、卡牌百科依次绘制。UI 仍通过 Canvas 2D/RenderCommand 与直接 Canvas 文本绘制完成，不使用单独 DOM 或 Pixi UI 容器。

## 3. 天气系统

### 3.1 天气类型

| 天气 | ID | 所属关卡 | 说明 |
|------|-----|---------|------|
| 晴天 | Sunny | 第2关 | 默认天气，阳光普照 |
| 下雨 | Rain | 第1关 | 大雨滂沱 |
| 下雪 | Snow | （预留） | 暴雪严寒 |
| 下雾 | Fog | （预留） | 浓雾弥漫，视野受限 |
| 夜晚 | Night | 第3、5关 | 夜幕降临 |
| 红雾 | RedMist | 第4关 | 赤红雾霾笼罩废土，远方火山将醒 |

### 3.2 视觉特效

| 天气 | 特效 |
|------|------|
| 晴天 | 左上角多层径向渐变太阳 + 5束斜向半透明阳光射线 + 低密度风线 |
| 下雨 | 双层斜雨：远层60滴、近层70滴；顶部乌云由28个大云团和40个小云团叠加；风线密度提高 |
| 下雪 | 40个固定槽位雪花（#ffffff），半径4-8px，初始横纵坐标随机分布，循环下落时从顶部重新进入，带正弦漂移 |
| 下雾 | 18个大雾团 + 30个小雾团，使用径向渐变软边、缓慢漂移和上下摆动 |
| 夜晚 | 70颗小星、35颗中星、8颗亮星 + 右上角三层光晕月亮 + 风线 + 暗角；可在关卡 `map.lighting.moonlight` 中启用棋盘外侧 bloom 和微光粒子 |
| 红雾 | 右上角巨大红色夕阳，使用外层红晕、暗红虹膜和黑色瞳孔构成“眼睛”剪影；天空 36 个暗红云/火山灰团快速横向移动；背景层绘制 3 层淡黑色渐变远山，山峰有大有小、层层叠叠，只有最大山顶有淡红光并冒出缓慢漂移的黑烟 |

当关卡图片背景有效开启时，程序化天空地标装饰必须隐藏，避免与图片背景中已有元素重复：晴天隐藏太阳与云朵，夜晚隐藏月亮，红雾隐藏远山与“眼睛”夕阳；关闭图片背景或美术资源总开关后恢复程序化表现。天气粒子、暗角、星空、雨雪雾等氛围叠加仍保留。

### 3.3 天气对游戏的影响

天气仅影响视觉氛围， **不影响战斗数值** 。

> 设计意图：避免玩家需要记忆天气对数值的影响，降低认知负担。天气纯粹用于烘托关卡氛围。

### 3.4 夜晚棋盘可读性

第3、5关使用夜晚天气时，棋盘主体必须保持可读。第5关深渊裂隙因主题色和夜晚遮罩叠加较暗，必须启用 `map.lighting.moonlight`：

- `enabled: true`
- `ambientAlpha: 0.16`，兼容旧配置；当前实现不使用该值绘制棋盘内部提亮层
- `bloomAlpha: 0.22`，用于棋盘向外 bloom 光晕和微光粒子强度

月光由 `BoardGlowSystem` 围绕 `RenderSystem.sceneOffsetX/Y/W/H` 绘制在棋盘外侧，只允许出现在棋盘外侧 bloom 区域，不进入棋盘内部。粒子必须保持稀疏、非均匀分布，不能沿棋盘边缘形成连续白色环形边框。月光不在棋盘内部叠加半透明铺色或径向遮罩，只通过外侧弱 bloom 营造被照亮的边缘氛围，不改变天气数值规则、单位属性或 UI 亮度。

### 4.3 动态环境生物

**飞鸟** （所有关卡共通）：

| 属性 | 值 |
|------|-----|
| 形状 | 简化V形（2条短线相交于中心）+ 小三角头 |
| 大小 | 8-14px翼展 |
| 颜色 | `#ffffff`（白色） |
| 数量 | 2-5只/场景 |
| 运动 | 沿贝塞尔曲线飞行，翅膀正弦拍动（3-5Hz） |
| 行为 | 循环路径：左→右→飞出后从另一侧重入→随机改变高度 |
| 层级 | LowAir层（z=4） |

翅膀拍动周期（1周期=2次拍动/秒）：
- 0-25%：翅膀展开↑，身体+2px
- 25-50%：翅膀收起↓，身体-2px
- 50-75%：翅膀展开↑，身体+2px
- 75-100%：翅膀收起↓，身体-2px
- 翅膀角度：30°（收起）→120°（展开）正弦循环

| 端点 | 主体资源 | 循环 FX 资源 | 层级 | 说明 |

## 6. 程序化单位美术系统

### 6.1 目标

塔、士兵、机关、敌人、法术弹道全部使用统一的程序化美术语言。基础几何体不再直接作为最终外观，而是作为“身体/底座”的一层，叠加部件、光影、动作和状态材质形成完整角色。

本系统优先保持战斗可读性：

- 轮廓必须清楚，32px 左右的小单位仍能区分阵营和职责。
- 同类别单位共享形体语言，不同职责通过部件差异区分。
- 状态效果必须叠加在单位外观之上，不覆盖核心识别色。
- 所有外观只影响表现，不改变碰撞、寻路、攻击、数值。

### 6.2 配置来源

外观配置由两级来源组成：

| 来源 | 优先级 | 用途 |
|------|--------|------|
| YAML `visualParts` | 高 | 关键单位的手工定制外观 |
| `proceduralArt` 生成器 | 低 | 为未配置 `visualParts` 的所有单位生成默认复合外观 |

`visualParts` 采用通用复合部件结构：

- `bodyParts`：身体装饰、底座、护甲、炮管、纹章、裂纹等。
- `weapon`：士兵近战/远程武器和可挥砍部件。
- `bobStyle`：`walking`、`floating`、`static` 三种待机/移动动效。

### 6.3 类别形体语言

| 类别 | 主体语言 | 必备部件 | 动作 |
|------|----------|----------|------|
| 塔 | 稳定底座 + 职责装置 | 底座、核心、炮管/水晶/线圈/巢穴 | 待机呼吸，攻击时装置轻微强调 |
| 士兵 | 圆形/盾形身体 + 头盔/职业徽记 + 武器 | 头盔、职业徽记、武器或工具 | 行走 bob，攻击挥砍/举弓 |
| 机关 | 贴地结构 + 触发部件 | 地盘、齿、坑、风口、符文 | 触发时展开/脉冲 |
| 地面敌人 | 生物身体 + 威胁轮廓 + 职责部件 | 角/甲壳/爪/机械板/核心高光 | 行走 bob，攻击压缩前冲 |
| 飞行敌人 | 悬浮身体 + 翼/气囊/推进器 | 翅膀或推进器、阴影弱化 | floating bob |
| Boss | 最大主体 + 多层光环/重护甲/主题核心 | 至少12个复合部件：投影、双层光环、主体、发光核心、护甲、角/翼/炮管/虫足等专属部件 | 行走或悬浮 bob + 呼吸脉冲，二阶段变色、强轮廓 |
| 法术/弹道 | 几何轨迹 + 粒子 | 拖尾、内核、外环 | 生命周期内缩放/淡出 |

### 6.4 视觉层次

单个单位按以下顺序绘制：

1. 地面阴影或低空投影。
2. 光环/蓄力/毒雾等低层状态。
3. 主体基础形状。
4. 复合 `bodyParts`。
5. 武器、炮管、翅膀等前景部件。
6. 徽记、核心、高光。
7. 选中框、精英金边、Boss 阶段轮廓。

部件可通过 `offsetX/offsetY/size/h/color/alpha/stroke/strokeWidth/rotation` 控制。可移动单位的 `offsetX` 随 `facing` 翻转；塔和机关默认为静态不翻转。

### 6.5 品质规范

- 每个可见单位至少包含 3 个视觉层：主体、部件、高光/描边。
- 玩家单位以冷色和高饱和职业色为主，敌人以暖色、暗色或主题色为主。
- 机关必须贴地，不允许使用与士兵相同的面部语言。
- 飞行单位必须使用 `floating` 动效，并通过翅膀、气囊或推进器说明高度。
- 精英敌人保留金色轮廓，Boss 必须是全场最高复杂度单位：比普通敌人拥有更多几何层、更多描边/光环、主题专属结构，并且不得退化为单形状或静态外观。
- 法术弹道优先用程序化拖尾、圆环、闪光、粒子，不引入贴图依赖。

### 6.6 塔外观系统

| 效果 | 触发时机 | 表现 |
|------|---------|------|
| 攻击波纹 | 敌人攻击动画期间 | 3个同心圆依次扩散并淡出 |

### 8.4 伤害飘字系统（Damage Numbers）

伤害飘字是战斗反馈的核心——每次攻击命中敌人时，在受击位置上方显示浮动伤害数字，让玩家直观感受到输出效果。系统通过 `DamageObserver` 模式自动捕获所有伤害事件。

| 属性 | 说明 |
|------|------|
| 触发时机 | 任何 `applyDamageToTarget()` 调用时自动触发（含弹道命中、AOE溅射、DOT跳伤） |
| 飘字样式 | Physical=橙色, Magic=紫色, True=金色, Heal=绿色, Critical=红色 |
| 动画 | 向上浮动 50±20 px/s，1.2s 存活，前半段保持不透明、后半段线性淡出 |
| 缩放入场 | 初始 0.6× → 0.3s 内放大到 1.0× |
| 水平偏移 | ±15px 随机偏移（避免重叠） |
| 渲染层 | 在场景实体之上、UI 之下绘制（post-render pipeline） |
| 实现 | `DamageNumberSystem` (ECS System) + `DamageNumber` 组件 (bitecs) |
| 位置 | `src/systems/DamageNumberSystem.ts` |

## 10. 胜利界面（Victory Screen）

### 10.1 定位

胜利界面是一个**全屏覆盖层**（z=1000），在战斗画面定格后以单层 3 阶段动画序列呈现。所有视觉、音频、故事参数由关卡 YAML 的 `victory` 配置节驱动。详见 `design/04-levels.md` §9。

### 10.2 渲染架构

```
VictoryScreenSystem (implement System)
  ├── 内部状态机: Phase1 → Phase2 → Phase3 → Done
  ├── Phase1 (0~1.2s): "恭喜你通关xxxx（关卡名）" 大字弹性弹出 (scale 2.2→1, elastic-out)
  │   └── 同步触发 ScreenShakeSystem (震幅6px, 持续0.5s, 频率25Hz)
  ├── Phase2 (1.2~4.2s): 彩带飘落 + 星星评定逐个闪烁 (每颗0.5s间隔)
  ├── Phase3 (4.2s+): 通关剧情标题浮现，正文以打字机效果逐字出现，不绘制底框
  │   └── 正文打完后显示 "点击继续" 提示，整屏点击继续
  └── 绘制: 在 UISystem.renderUI() 之后，直接写 Canvas 2D
```

通关标题使用 `typography.titleColor` 渐变填充，baseSize=64px，内容为“恭喜你通关xxxx（关卡名）”。光晕随 alpha 从 0 → 24px 渐变。屏幕震动在 `main.ts:handleVictory()` 中通过 `ScreenShakeSystem.triggerShake()` 触发。

Defeat 模式使用同一套全屏覆盖层，不再显示旧的 `UISystem` 失败覆盖层和失败故事面板。失败时直接显示失败剧情标题，剧情正文在同一层以打字机效果逐字出现，文字直接浮现在画面上且不带底框；正文打完后显示“点击返回”，整屏点击返回关卡选择。

### 10.3 彩带粒子系统

```typescript
interface ConfettiParticle {
  x: number; y: number;
  vx: number; vy: number;
  rotation: number; rotSpeed: number;
  width: number; height: number;
  color: string;
  alpha: number;
  life: number; maxLife: number;
  shape: 'ribbon' | 'petal' | 'sparkle' | 'fragment';
}
```

| shape | 绘制方式 | 使用场景 |
|-------|---------|---------|
| `ribbon` | 细长矩形（w=3px, h=8~14px），带旋转 | 通用彩带 |
| `petal` | 椭圆（rx=3px, ry=6px），带旋转 | 自然主题（草原） |
| `sparkle` | 小圆（r=2~4px），内发光 | 魔法/庆典 |
| `fragment` | 不规则三角（3个顶点随机偏移） | 机械/废墟主题 |

粒子初始速度由 `burst` 决定：
- `top_fall`: vx=随机±spread, vy=100~200（下落加速）
- `bottom_rise`: vx=随机±spread, vy=-150~-80（上抛减速）
- `explosion_center`: 从屏幕中心径向发射
- `both_sides`: 左半屏向右、右半屏向左发射

每帧更新：位置 += 速度 × dt，旋转 += 旋转速度，alpha = 1 - life/maxLife（线性淡出）。

### 10.4 背景滤镜类型

| filter | 实现方式 |
|--------|---------|
| `gray_tint` | 战斗画面叠 80% 灰度（ctx.globalCompositeOperation = 'saturation'，设置 saturation 为 0），再叠加半透明暗色 |
| `rain_to_sunny` | 灰度定格 → 从顶部向底部渐变填充 `gradient` 三色 → 右上角绘制金色光晕（`createRadialGradient`）随时间增大 |
| `heat_dissipate` | 棕黄 tint 叠加 + 水平波浪线（正弦曲线）随时间降低振幅 → 残骸粒子（小三角/椭圆）从屏幕随机位置生成并缓慢下落 |
| `dawn_break` | 暗角 tint → 屏幕中央绘制垂直裂缝状白光（两条锯齿线条+中心径向渐变），随时间扩展 → 蝙蝠剪影（V 形）从裂缝向屏幕边缘飞出 |

## 11. 关卡入场动画（Level Intro）

> **v4.1 新增** — 关卡启动的仪式感动画序列，在部署阶段之前播放。
> **依赖**: 04-levels（关卡配置）、06-technical（系统管线）
> **实现**: `src/systems/LevelIntroSystem.ts`

### 11.1 设计目标

玩家选择关卡后不直接进入冰冷的部署画面，而是通过一段 **5阶段仪式感动画** 建立"战场正在形成"的沉浸感：

1. 普通场景地格随机从天而降，所有格子先显示为同主题普通场景地格
2. 场景装饰显现
3. 水晶位置的普通场景地格被水晶地格替换，水晶实体同步显现
4. 出生口位置的普通场景地格被出生口地格替换，传送门主体与特效同步显现
5. 敌军进攻路线按 `pathGraph` 顺序从出生口向水晶铺开，路径格逐格替换普通场景地格

动画结束后无缝衔接第一波倒计时。

### 11.2 动画阶段时间线

```
t=0.00          t=1.90     t=2.50     t=3.10     t=3.60        t=按路径长度       +5s
  │  Phase 1      │ Phase 2  │ Phase 3  │ Phase 4  │  Phase 5     │  Countdown    │ Wave 1
  │普通地格掉落    │ 装饰显现 │ 水晶替换 │ 出生口替换│  路径替换    │  5秒准备      │ 开始！
  │(1.2s+0.5s交错)│ (0.6s)  │ (0.6s)   │ (0.5s)   │  20ms/格     │              │
  ▼               ▼          ▼          ▼          ▼              ▼              ▼
```

| 时间 | 阶段 | 行为 |
|------|------|------|
| 0.00s | Phase 1 开始 | 所有棋盘格按同主题普通场景地格渲染，从屏幕上方（-200~-600px 随机）坠落，装饰物不可见 |
| 0.00~1.90s | 普通地格掉落 | 瓦片按二次缓入（ease-in）坠落到目标位置，延迟+速度差异形成自然层次感；即使配置中是 path/spawn/base/blocked，此阶段也显示普通场景地格 |
| 1.90s | Phase 1→2 | 最后一个瓦片着地，装饰物开始显现 |
| 1.90~2.50s | 装饰显现 | 树、岩石、灌木、草丛等静态装饰物 alpha 从 0→1 淡入（0.6s）；云朵、飞鸟同步显现 |
| 2.50s | Phase 2→3 | 场景装饰就绪，水晶地格开始替换 |
| 2.50~3.10s | 水晶替换 | 水晶位置的普通场景地格被 `TileType.Base` 水晶端点地格淡入替换；水晶实体 alpha 从 0→1 同步显现 |
| 3.10s | Phase 3→4 | 水晶完全显现，出生口开始替换 |
| 3.10~3.60s | 出生口替换 | 所有 Spawn 位置的普通场景地格被 `TileType.Spawn` 出生口端点地格淡入替换；传送门主体与循环 FX alpha 从 0→1 淡入 |
| 3.60s | Phase 4→5 | 出生口就绪，路径开始铺展 |
| 3.60s 起 | 路径替换 | 从每个 Spawn 出发沿 `pathGraph` 到水晶，路径位置按顺序每 20ms 替换 1 个 `TileType.Path` 地格；已替换地格保持路径美术资源，未替换地格仍为普通场景地格 |
| 按路径长度结束 | 动画完成 | 所有路径地格铺完，恢复正常渲染 |
| 动画完成后 5s | 部署倒计时 | 玩家可拖卡布阵 |
| 倒计时结束 | Wave 1 | 第一波敌人涌出 |

### 11.3 瓦片掉落物理

- **起始 Y**: `-200 + random(-400)`（棋盘上方 200~600px 视口外）
- **缓动函数**: `progress²`（二次 ease-in），模拟重力加速
- **随机延迟**: 每个瓦片 `random(0, 0.5)` 秒，首尾交错形成"层层叠落"效果
- **目标位置**: 同正常棋盘渲染坐标（`col * ts + ts/2 + ox`, `row * ts + ts/2 + oy`）
- **视觉**: 瓦片 alpha=1.0，使用 `tile_<theme>_buildable.png` 普通场景地格美术资源；资源未加载时才回退同主题配置色，z=0

### 11.4 水晶显现

- 水晶实体已在 `initBattle()` 中创建，但入场动画 Phase 3 前 alpha 必须为 0
- 水晶地格先从普通场景地格替换为 `tile_<theme>_path_endpoint_crystal.png`
- **Alpha**: 0→1 线性过渡（0.6s）
- **Scale**: 0.3→1.0 弹性缓出（overshoot），增强"降世"仪式感
- 水晶的 Visual 组件数据在动画期间被 `LevelIntroSystem` 临时覆写

### 11.5 传送门显现

- Spawn 格子先从普通场景地格替换为 `tile_<theme>_path_endpoint_spawn.png`
- 传送门主体使用 `objective_spawn_portal.png`，循环 FX 使用 `fx_spawn_portal_loop_0.png`
- 入场动画期间，出生口地格、传送门主体和传送门 FX alpha 从 0→1 淡入
- 传送门的旋转菱形环从静止开始逐渐加速（0→正常速度，0.5s ease-in）
- 暗红脉冲效果同步增强

### 11.6 路径铺展（按 pathGraph 逐格替换）

- 从每个 Spawn 节点出发，按 `pathGraph.edges` 方向沿路走向 `crystal_anchor`
- 路径瓦片按 `{row, col, distance}` 排序（距离近的先揭示）
- 每格间隔 20ms，产生清晰的路线铺开节奏
- 已揭示路径位置从普通场景地格替换为 `tile_<theme>_path.png`
- 未揭示路径位置继续显示普通场景地格，不画路径高亮遮罩
- 多 Spawn 点路径合流时同一格只替换一次，取最短距离顺序

### 11.7 路径替换算法

```
输入: map.pathGraph.nodes / map.pathGraph.edges
输出: pathRevealOrder[{row, col, distance, sequence}][]

建立 nodeById 与 outgoing(edge.from -> edge.to[])
对每个 role=spawn 的节点:
  queue.enqueue({node: spawnNode, distance: 0})
  while queue 非空:
    current = queue.dequeue()
    对 current.node 的每条 outgoing edge:
      target = nodeById[edge.to]
      steps = interpolateEdge(current.node, target)
      对 steps 中每个格子:
        若 tiles[row][col] === Path:
          用最短 distance 记录到 entriesByTile
      queue.enqueue({node: target, distance: 当前路径距离})

按 distance 升序、sequence 升序排序 entriesByTile → pathRevealOrder[]
```

### 11.8 实现架构

```
LevelIntroSystem (新 System，PHASE_CREATION 阶段注册)
  ├── state: IntroPhase { TilesFalling, DecorAppear, CrystalAppear, SpawnAppear, PathReveal, Complete }
  ├── timer: 阶段内计时器
  ├── tileAnimData[]: 每个瓦片的起始/目标/当前 Y + 延迟
  ├── pathRevealOrder[]: 按 pathGraph 排序后的路径瓦片列表
  ├── pathRevealIndex: 当前揭示进度
  │
  ├── update(): 推进状态机 → 更新各相位的动画参数
  ├── render(): 直接绘制到 Canvas 2D context
  │     ├── Phase 1: 按 currentY 绘制普通场景地格
  │     ├── Phase 2: 绘制普通场景地格 + 装饰淡入
  │     ├── Phase 3: 水晶地格替换 + 水晶实体淡入
  │     ├── Phase 4: 出生口地格替换 + 传送门主体/FX 淡入
  │     └── Phase 5: 每 20ms 将一个路径位置替换为路径地格
  │
  └── onComplete(): 
        ├── RenderSystem.introActive = false (恢复正常棋盘渲染)
        └── WaveSystem.startAutoCountdown(5) (启动部署倒计时)
```
