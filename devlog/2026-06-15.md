## 技术咨询：游戏是否使用图集系统

### 结论

当前项目已经在运行时代码层支持图集 manifest 和图集帧绘制，但仓库 `public/art` 下尚未提交实际图集产物，因此实际运行时默认仍会回退到独立图片资源。

### 判断依据

- `src/utils/imageCache.ts` 会尝试读取 `/art/atlases/index.json`，并将旧资源路径映射到图集图片和源矩形。
- `getLoadedImageFrame()` 命中 manifest 时返回 `atlasId`、`source` 和图集图片；未命中或图集加载失败时回退独立图片。
- `drawImageFrame()` / `drawLoadedImage()` 已支持使用源矩形裁剪绘制图集帧。
- `src/utils/imageCache.test.ts` 已覆盖图集命中、单图回退、源矩形绘制和美术资源关闭场景。
- `design/10-art-atlas-pipeline.md` 定义了图集分包、manifest 格式、加载策略和验收标准。
- 当前 `public/art` 没有 `atlases` 目录，也没有实际 `index.json`、图集 PNG 或图集 JSON 产物。

### 结论表述

准确说法是：项目“有图集系统的运行时接入层和设计规范”，但“当前资源包还没有真正启用图集产物”。要让游戏实际使用图集，需要按 `design/10-art-atlas-pipeline.md` 生成并提交 `/public/art/atlases/index.json` 及对应图集图片。

## 分析：激光塔强度过高

### 需求

用户反馈激光塔数值太强，接近无敌，需要分析削弱方向。

### 调查结论

- 激光塔配置为 `atk: 7`、`attackSpeed: 0.5`、`range: 250`、造价 125，但实际光束伤害没有使用自身 `atk`。
- `AttackSystem.doLaserAttack` / 导出的 `doLaserAttack` 都把每束伤害上限写死为 `TOWER_CONFIGS[TowerType.Arrow].atk * 3`，当前等于 30。
- 光束持续 5 秒，攻击冷却 2 秒，稳定期同一座激光塔会叠 2-3 批光束。
- `LaserBeamSystem` 每 0.1 秒结算一次，单束全程约 775 总伤害，折合约 155 DPS；叠层后 L1 稳态约 387 DPS。
- 等级 3 起变为 2 束，等级 5 为 3 束，多目标输出会按束数线性放大。
- 升级给激光塔加的 `atkGrowth` 会写入 `Attack.damage`，但光束 `maxDamage` 不读该值，因此升级数值与实际伤害契约脱节。

### 建议

- 优先把光束上限改为读取激光塔自身有效攻击力，而不是箭塔固定攻击力。
- 将持续时间压到不超过冷却，或显式禁止同一座塔的光束叠层。
- 保留“持续锁敌、越照越痛”的职业特征，但降低每束结算频率或总伤害预算。
- 若需要保留高阶多束，应把 L3/L5 多束作为主要成长点，基础 L1 强度必须明显下调。

## 修复：巨型史莱姆死亡后不分裂

### 问题

巨型史莱姆 HP 归零后没有生成中型史莱姆。分裂逻辑已在 `BossSystem` 中实现，但主循环里 `LifecycleSystem` 先运行，会对非最终分裂层做早退等待；如果 BossSystem 没有在生命周期死亡判定前接手，史莱姆会停留在 0 HP 状态，表现为没有分裂。

复测后发现还存在配置兜底问题：静态 Boss 配置只声明了 `isBoss`，部分条目未声明 `bossType`。当运行路径未被 YAML 注入覆盖，`WaveSystem` 会把它们生成为数据驱动 Boss，导致 `BossSystem` 的专属 Boss 分支不执行。除巨型史莱姆外，虫族女王、路西法、超级机器人、深渊领主也需要同类兜底。

### 改动

| 文件 | 改动 |
|------|------|
| `src/main.ts` | 调整战斗系统注册顺序，让 `BossSystem` 先于 `LifecycleSystem` 处理巨型史莱姆非最终死亡分裂 |
| `src/data/gameData.ts` | 为 5 个静态 Boss 配置补充 `bossType`；`giant_slime` 额外补 `splitCount: 0` |
| `src/systems/WaveSystem.ts` | 增加 Boss 敌人 ID 到专用 `BossType` 的兜底映射，避免配置注入缺失时退化成数据驱动 Boss |
| `src/systems/BossSystem.test.ts` | 增加主循环管线顺序回归测试，验证 0 HP 巨型史莱姆在一帧更新后分裂为 2 个中型史莱姆 |
| `src/systems/__tests__/WaveSystem.spawn.test.ts` | 增加从波次生成 5 个 Boss 后均挂载专用 `BossType` 的回归测试，并保留巨型史莱姆分裂测试 |

### 验证结果

- `npm test -- src/systems/BossSystem.test.ts src/systems/LifecycleSystem.test.ts` 通过，2 个测试文件、22 个用例。
- `npm run typecheck` 通过。
- `npm test -- src/systems/__tests__/WaveSystem.spawn.test.ts src/systems/BossSystem.test.ts src/systems/LifecycleSystem.test.ts` 通过，3 个测试文件、47 个用例。
