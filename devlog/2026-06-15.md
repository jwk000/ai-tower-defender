## 技术咨询：游戏是否使用图集系统

### 结论

当前项目已经在运行时代码层支持图集 manifest 和图集帧绘制，但仓库 `public/art` 下尚未提交实际图集产物，因此实际运行时默认仍会回退到独立图片资源。

### 判断依据

- `src/utils/imageCache.ts` 会尝试读取 `/art/atlases/index.json`，并将旧资源路径映射到图集图片和源矩形。
- `getLoadedImageFrame()` 命中 manifest 时返回 `atlasId`、`source` 和图集图片；未命中或图集加载失败时回退独立图片。
- `drawImageFrame()` / `drawLoadedImage()` 已支持使用源矩形裁剪绘制图集帧。
- `src/utils/imageCache.test.ts` 已覆盖图集命中、单图回退、源矩形绘制和美术资源关闭场景。
- `design/10-art-atlas-pipeline.md` 定义了图集分包、manifest 格式、加载策略和验收标准。
- 当前 `public/art` 没有 `atlases` 目录，也没有实际 `index.json`、图集 PNG 或图集 JSON 产物。

### 结论表述

准确说法是：项目“有图集系统的运行时接入层和设计规范”，但“当前资源包还没有真正启用图集产物”。要让游戏实际使用图集，需要按 `design/10-art-atlas-pipeline.md` 生成并提交 `/public/art/atlases/index.json` 及对应图集图片。

## 功能：生成图集并接入运行时图集加载

### 需求

为当前 `public/art` 资源生成实际图集产物，并让运行时代码显式引用图集加载路径，减少单位、地格、FX、UI 和图标资源的独立图片请求。

### 改动文件

| 文件 | 改动 |
|------|------|
| `scripts/build-art-atlases.mjs` | 新增图集构建脚本，基于 Pillow 生成 PNG 图集和 manifest |
| `package.json` | 新增 `npm run build:atlases` 命令 |
| `public/art/atlases/` | 新增 19 张图集及 JSON manifest，覆盖 503 个资源帧 |
| `src/utils/imageCache.ts` | 新增显式图集索引加载、图集图片预热和测试状态查询接口 |
| `src/main.ts` | 游戏启动时预加载图集索引，进关时预热图集图片 |
| `src/utils/imageCache.test.ts` | 增加预加载 atlas index 后通过图集帧解析资源路径的回归测试 |

### 生成结果

- 单位状态帧：拆成 `units_01` 到 `units_08`，覆盖 376 张 256×256 单位图。
- 主题地格：按 `meadow/desert/castle/wasteland/abyss` 各生成一张主题图集，运行时帧尺寸降为 256×256。
- 战斗通用：`fx_objectives` 覆盖 FX 和水晶/传送门目标点资源。
- UI/Icon：`icons_common` 覆盖 Buff 与 card-icon；`ui_common_01` 到 `ui_common_04` 覆盖 UI 基础件。
- `public/art/atlases/index.json` 作为运行时统一入口，manifest key 保留旧资源路径，业务代码无需改配置路径。

### 验证结果

- `npm run build:atlases` 通过，生成 19 张图集、503 个帧映射。
- `npm test -- src/utils/imageCache.test.ts src/utils/artAssets.test.ts` 通过，2 个测试文件、11 个用例。
- `npm test -- src/utils/imageCache.test.ts src/systems/RenderSystem.crystal.test.ts src/systems/UISystem.test.ts` 通过，3 个测试文件、23 个用例。
- `npm run typecheck` 通过。
- `npm run build` 通过；保留 Vite 既有大 chunk 体积警告。

## 重设计：按 MDA 理论重平衡全关卡数值

### 需求

按 MDA 理论重新设计 5 个关卡的数值，覆盖卡池、波次奖励、敌人数量与高威胁单位并发压力，保证在最坏金币掉落情况下玩家仍有资源构建基础防线，不出现资源死局。

### 改动文件

| 文件 | 改动 |
|------|------|
| `design/04-levels.md` | 新增 MDA 数值重设计计划，记录最坏预算口径、调整原则和每关策略 |
| `src/config/levels/*.yaml` | 调整 5 关初始金币、波次奖励、高威胁敌人数量与第4关卡池 |
| `src/editor/state/modelToLevelConfig.ts` | 透传 YAML 波次 `reward` 到运行时 `WaveConfig.reward` |
| `src/config/levels.test.ts` | 增加敌人引用、波次奖励透传、最坏金币下限预算校验 |

### 关键决策

- 敌方黑暗牧师统一使用 `dark_priest`，避免关卡波次误引用玩家祭司 `priest`。
- 预算按 `初始金币 + 击杀金币下限(80%) + 波次奖励` 计算，并把每波额外随机精英纳入最坏金币下限。
- 降低第3-5关后段牧师/坦克/无人机/巨型机器人等高威胁单位的多路叠压，保留主题压力但给玩家可恢复窗口。

## 设计：补全低空单位与防空对策说明

### 需求

确认每一关哪些敌人属于低空单位，以及哪些塔/单位能攻击低空单位。现有文档中 LowAir 信息分散在单位表和关卡波次里，缺少统一白名单和关卡验收表。

### 改动文件

| 文件 | 改动 |
|------|------|
| `design/03-units.md` | 新增低空攻击规则，明确 LowAir 对空白名单与不可对空单位，并为相关塔/士兵补充“可攻击低空”标注 |
| `design/04-levels.md` | 新增低空单位与防空对策总览，按关卡列出 LowAir 敌人、出现波次、初始卡池对策和设计结论 |

### 关键结论

- LowAir 敌人为：吸血蝗虫、吸血蝙蝠、无人机、飞机。
- 可稳定攻击 LowAir 的单位为：箭塔、弩塔、激光塔、蝙蝠塔、电塔、弓手。
- 导弹塔定位为地面战略爆炸，不处理 LowAir；炮塔、冰塔、火塔、毒塔、法师和地面机关也不命中 LowAir。
- 第2关从第3波引入低空教学，第3-5关均需要卡池或抽牌保底提供防空能力。

### 验证结果

- 已通过文档和配置检索确认 `locust`、`vampire_bat`、`plane`、`drone` 在设计/配置中对应 LowAir 敌人。
