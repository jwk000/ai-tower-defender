## 排查：第4关敌人半路停住

### 现象

第4关战斗截图中，大批敌人在中路交战区域聚集，看起来没有继续向水晶推进。

### 排查结果

- 第4关 `pathGraph` 三条出生路径均有完整后继，`spawn_0`、`spawn_1`、`spawn_2` 都连接到 `crystal_anchor`，未发现路径断点。
- MovementSystem 在敌人 FollowPath 移动前会先检查攻击：只要敌人有 `Attack` 组件且范围内存在士兵或塔，就执行攻击并把 `Movement.currentSpeed` 置为 0，本帧不推进路径。
- 攻击动画期间 `Visual.attackAnimTimer > 0` 时同样会停止移动。截图中的弓手位于中路交战点附近，第4关敌人配置包含机器狗、无人机、飞机、油罐、坦克等带攻击范围的单位，因此会被附近玩家单位吸住反复攻击。

### 结论

当前表现不是第4关路径配置卡死，而是敌人攻击玩家单位时主动停下。若产品目标是普通敌人持续冲水晶，需要先调整设计：例如仅特定敌人可停下拆塔/打士兵，或给普通敌人增加“忽略玩家单位继续推进”的行为标记。

## 修复：第4关无人机群停在半路

### 现象

第4关无人机/飞行敌人在路径中段或合流点附近没有攻击动作，但停在原地不继续前进。

### 根因

`drone` 属于 `EnemyFlockMember` 鸟群/无人机群敌人，移动走 `processFlockEnemy`。该逻辑会给成员施加 `anchorOffsetX/Y` 编队偏移，让成员追随“路径点 + 偏移”的目标点；但路径推进判定只检查实体是否靠近原始路径点。当偏移距离大于 `FLOCK_PATH_ADVANCE_RADIUS` 时，成员会稳定停在偏移目标附近，却永远不满足“靠近原始 waypoint”的推进条件。

### 改动

- `MovementSystem.processFlockEnemy` 的路径推进判定同时接受原始 waypoint 与带编队偏移的目标点。
- 增加回归测试：LowAir 鸟群敌人靠近带偏移的编队目标点时必须推进 `pathIndex` 并能结算到达水晶。

### 验证

- `npm test -- src/systems/MovementSystem.test.ts` 通过，16 个用例。
- `npm run typecheck` 通过。

## 修复：弓手/法师在场时敌人原地停住

### 现象

场上存在弓手、法师等远程士兵后，敌人会停在路径中段，不继续前进，也没有明显攻击我方单位。

### 根因

敌人被位移到路径段外后会触发 `MovementSystem.tryPathRecovery`。旧逻辑恢复到最近 waypoint 后本帧直接 `continue`，把 `Movement.currentSpeed` 置为 0。远程士兵追击/站位会让敌人与路径恢复持续交替出现，导致敌人每帧都只恢复、不推进，视觉上表现为全体原地停住且没有攻击动作。

### 改动

- 路径恢复后刷新 `pathIndex` 和当前位置，并在同一帧继续执行路径推进。
- 更新 path recovery 回归测试：敌人被拉回路径后应立即继续移动，`currentSpeed` 保持大于 0。

### 验证

- `npm test -- src/systems/__tests__/MovementSystem.soldier.test.ts src/systems/MovementSystem.test.ts src/systems/SoldierAISystem.test.ts` 通过，73 个用例。
- `npm run typecheck` 通过。

### 改动

## 修复：嘲讽目标超出攻击范围时敌人停步

### 现象

敌人带有嘲讽状态时，如果嘲讽源仍存活但不在敌人的攻击范围内，旧逻辑会让敌人本帧停止移动；在战场拥挤或远程单位牵引场景下，容易表现为敌人停住但不攻击。

### 根因

`MovementSystem.processTauntTimer()` 只校验嘲讽源是否存活和计时是否有效，返回 true 后主流程无条件停止敌人移动。随后 `processEnemyAttack()` 如果发现嘲讽源超出攻击范围，会返回 false，但移动已经被跳过。

### 改动

- 嘲讽计时仍正常递减和锁定目标。
- 只有 `processEnemyAttack()` 真正执行攻击时，敌人才因嘲讽停步。
- 新增回归测试：被嘲讽但攻击距离不够的敌人继续沿路径移动。

### 验证

- `npm test -- src/systems/SkillSystem.test.ts src/systems/MovementSystem.test.ts src/systems/__tests__/MovementSystem.soldier.test.ts` 通过，36 个用例。
- `npm run typecheck` 通过。

## 检查：死亡动作播放链路

### 结论

死亡动作播放链路存在：单位死亡后，`LifecycleSystem` 会创建带 `DeathEffect` 的临时实体，并通过 `deathSpriteRegistry` 记录原单位的 art id；`RenderSystem.getUnitSpriteState()` 读取到该映射后会优先选择 `death` 状态，按 `DeathEffect.elapsed / duration` 切换 `death_0/1` 帧。

### 改动

- `DeathEffectSystem` 从 `eid = 0` 开始扫描死亡特效，避免 0 号实体上的死亡效果被漏计时。
- `LifecycleSystem.test.ts` 增加回归测试，确认普通敌人死亡效果继承原单位 `death` 动作资源，并在死亡效果结束后清理映射。

### 验证

- `npm test -- src/systems/LifecycleSystem.test.ts` 通过，7 个用例。
- `npm run typecheck` 通过。
- 资源检查：`public/art/units` 下 46 个单位均存在 `death_0` 和 `death_1`，缺失数为 0；弓手和哥布林死亡帧均存在。
